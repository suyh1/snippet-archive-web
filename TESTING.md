# 项目通用测试检查方法（中文）

本文件是跨项目可复用的测试检查方案，适用于前端、后端、全栈交互场景。  
目标：避免“看起来没问题但实际上有缺口”的假通过。

## 0. 极简执行清单

每次任务默认按以下条目执行，可直接打勾：

1. 我是否明确了改动影响的用户行为（而不只是代码点）？
2. 我是否为本次改动准备了目标回归测试（先红后绿）？
3. 我是否覆盖了 click 流程（含取消/回退路径）？
4. 我是否覆盖了 keyboard 流程（`Enter`、`Esc`、连续输入）？
5. 我是否覆盖了 focus/blur 流程（焦点切换、失焦提交/取消）？
6. 我是否验证了状态迁移（UI 文案/顺序/选中态/持久化）？
7. 若涉及布局，我是否做了几何断言（容器-视口、子-父、文档溢出）并覆盖稀疏态+稠密态？
8. 若涉及危险操作（删除/移动/覆盖），我是否验证了取消与确认两条路径，并做了持久化校验？
9. 若涉及网络请求，我是否验证了关键请求的 method/url/status 和必要的 header/body 组合？
10. 若涉及确认/警告交互，我是否验证了使用项目自定义组件（而不是 `window.alert/confirm/prompt`）且风格一致？
11. 我是否已运行并确认：目标回归、全量测试、typecheck、build？

## 1. 总原则

1. 不以单一信号判断通过（例如只看 build 成功）。
2. 任何用户可感知行为变更，都必须做行为级验证（真实输入与状态变化）。
3. 任何 bug 修复，都必须有可复现原问题的回归测试。
4. 先证据后结论：没有最新命令输出，不得声称“已修复/已通过”。
5. 发现预期不一致时，继续迭代直到行为与产品意图一致。

## 2. 标准检查层级（从小到大）

1. `单元/组件层`：最小行为、边界、错误分支。
2. `集成层`：模块之间数据流与状态同步。
3. `行为层（E2E）`：真实点击、键盘、焦点、弹窗、网络。
4. `全量门禁`：完整测试集 + typecheck + build。

## 3. 行为级检查清单（必做）

每个交互改动，至少覆盖以下四类：

1. Click 流程：点击触发、二次点击、取消路径。
2. Keyboard 流程：`Enter`、`Esc`、连续输入、快捷键（若有）。
3. Focus/Blur 流程：焦点切换、失焦提交/取消、错误态恢复。
4. State Transition：UI 文案、列表顺序、选中态、持久化结果。

## 4. 布局与溢出检查（高频漏检点）

涉及全高布局、分栏、分页/分段/虚拟列表时，必须用几何断言而非纯肉眼：

1. 容器与视口关系：主容器高度对齐视口。
2. 子容器与父容器关系：底部/顶部/可用区边界对齐。
3. 文档溢出关系：`documentElement.scrollHeight`、`body.scrollHeight` 不应异常超出视口。
4. 必测两种数据态：
   - 稀疏态（空/少量数据）
   - 稠密态（大量数据）
5. 建议显式容差：
   - 对齐误差：`<= 2px`
   - 视口高度比较：`<= 4px`

## 5. 删除/危险操作检查（高风险）

删除、移动、覆盖等操作，至少要验证：

1. 取消路径：用户取消后数据不变。
2. 确认路径：用户确认后 UI 与后端都变化。
3. 持久化校验：通过 API 或数据层再次确认结果，而不只看页面。
4. 错误分支：失败时有可见反馈且状态不脏。

## 6. 网络与协议检查（常见真实故障）

当行为依赖 HTTP 请求时，建议在 E2E 中补充：

1. 关键请求状态码断言（成功/失败）。
2. 关键请求方法与 URL 断言（防路由/参数错误）。
3. 关键请求头与 body 断言（例如无 body 的 `DELETE` 不应误带 JSON Content-Type）。
4. 若 UI 未更新，先区分：
   - 后端未成功（请求失败）
   - 后端成功但前端未同步（状态管理/重渲染问题）

## 6.1 风格一致性与系统弹窗禁用

确认/警告类交互除了功能正确，还要检查视觉与交互一致性：

1. 禁止在主业务交互链路中直接使用 `window.alert/confirm/prompt`。
2. 必须使用项目统一风格的自定义 Dialog/Toast 组件。
3. E2E 要有“反向保护断言”：
   - 监听浏览器原生 `dialog` 事件
   - 若捕获到原生弹窗，测试直接失败
4. 对删除等危险操作，验证分级策略是否执行到位：
   - 低风险项（如普通文件）：轻量确认（如行内二次确认）+ 可撤销
   - 高风险项（如文件夹/工作区）：明确的模态确认

## 7. 回归测试编写要求

1. 回归测试必须先红后绿（先失败再修复）。
2. 断言应对准行为结果，不绑死实现细节。
3. 单测与 E2E 分工明确：
   - 单测：小而快，覆盖分支逻辑
   - E2E：关键业务链路闭环

## 8. 完成前强制门禁

合并/交付前，必须全部通过：

1. 改动相关的目标回归测试
2. 全量测试
3. Typecheck
4. Build

## 9. 交付汇报模板（建议）

1. 本次验证了哪些交互流（click / keyboard / focus-blur / state）。
2. 新增或更新了哪些回归测试（文件路径）。
3. 全量验证命令与结果（test/typecheck/build）。
4. 遗留风险（若有）与后续建议。

## 10. 与仓库规则的关系

1. 本文是通用方法学。
2. 具体仓库规则以 `AGENTS.md` 为准。
3. 当两者冲突时，执行“更严格”的那一条。
