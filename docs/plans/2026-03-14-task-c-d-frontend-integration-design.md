# Task C + Task D 设计文档（前端接入真实后端 + 文件树主链路）

## 1. 目标

在现有 NestJS Workspace API v1 的基础上，完成前端可操作主链路：

1. Task C：前端从静态占位页升级为 API 驱动，完成工作区列表/打开/新建/删除。
2. Task D：接入文件树、文件夹、拖拽移动，并确保 `order` 与 `path` 规则一致。

## 2. 方案选型

已选方案：`API Client + Pinia + 树结构 + 拖拽移动`。

理由：

- 与后续扩展（编辑器、文件夹操作、批量动作）兼容性最佳。
- 把网络层、状态层、UI 层解耦，避免组件直接请求导致维护成本上升。
- 可在不引入重型依赖的情况下，用原生拖拽先打通主链路。

## 3. 架构设计

### 3.1 前端分层

- `src/api`: 统一 `fetch` 封装与 workspace 领域 API。
- `src/stores`: Pinia 状态管理（工作区列表、当前工作区、文件列表、加载态）。
- `src/features/workspace`: 视图组件（Library/Workspace 双态、文件树、操作区）。
- `src/utils`: 文件路径与树结构转换工具。

### 3.2 后端扩展（支持 Task D）

在现有 file CRUD 基础上新增 `move` 语义接口：

- `PATCH /api/workspaces/:workspaceId/files/:fileId/move`

职责：

- 接收目标路径与目标顺位。
- 对文件夹移动时同步更新子孙节点路径。
- 统一重排受影响目录的 `order`，保证顺序连续。

## 4. 数据与规则

### 4.1 路径规则

- 所有文件/文件夹路径必须以 `/` 开头。
- 每个工作区内 `path` 唯一。
- 文件夹路径移动时，后代路径按前缀替换。

### 4.2 顺序规则

- `order` 表示同级节点顺位（同父目录）。
- 创建/移动/删除后触发归一化：同级从 `1..N` 连续。
- 前端拖拽提交目标目录与目标顺序，后端为最终一致性裁决层。

## 5. 前端交互设计

### 5.1 Library / Workspace 双态

- `Library`：展示工作区列表和创建入口。
- `Workspace`：展示当前工作区标题、文件树、文件/文件夹创建与拖拽移动。

### 5.2 主链路

- 打开应用自动加载工作区列表。
- 点击工作区进入 Workspace 视图并加载文件列表。
- 可新建/删除工作区，删除后自动回退到 Library 或相邻工作区。
- 文件树支持：
  - 新建文件
  - 新建文件夹
  - 拖拽文件/文件夹到目标目录
